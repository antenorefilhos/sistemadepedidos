1 🌟 Pontos Fortes

  • Stack moderna e bem escolhida: Next.js 16 App Router + React 19 + Tailwind v4 (arquitetura CSS-first @theme) + DaisyUI v5 + Supabase/Postgres. Nenhuma dessas escolhas está datada ou é questionável para o porte do negócio.
  • Uso correto de Server Components onde importa mais: produtos/[slug]/page.js e receitas/[slug]/page.js fazem fetch direto no servidor via getSupabase(), com generateMetadata dinâmico bem implementado (título, descrição, Open Graph a partir do produto/receita real).
  • SellerReferralTracker usa useSearchParams corretamente isolado em <Suspense fallback={null}> — padrão certo para não bloquear SSR.
  • Pipeline de upload de imagem já otimizado: conversão automática para WebP via sharp no backend, com resize por tipo (800px produto vs 1400px genérico) — compensa parcialmente o images.unoptimized: true.
  • RLS parcialmente implementado nas tabelas centrais do domínio de vendas (products, orders, order_items, sellers, categories) — a base do modelo de segurança existe, só não está completa.
  • Painel admin: depois do trabalho desta sessão, hoje tem um kit de UI compartilhado (Modal, Toast, ConfirmDialog, StatusBadge, Tag, StatCard, DataTable, ImageUploadField), autenticação/fetch centralizados (adminFetch), tipografia unificada, nenhum arquivo com mais de ~530 linhas, e dois bugs de negócio reais corrigidos. É a área mais saudável do projeto hoje.
  • JSON-LD na home (FoodEstablishment com endereço, geo, horário) — presente e correto, só não replicado em outras páginas.

2 ⚠️ Ajustes Imediatos (Alta Prioridade)

Ordenado por severidade real, não por facilidade de correção.

  1. Senha de administrador real, hardcoded como fallback, commitada no histórico do git. A string 'Aef@1945*' aparece idêntica em src/app/api/admin/{customers,migrate,orders,reviews}/route.js:9 e src/app/api/admin/upload/route.js:96. Não é uma senha de exemplo genérica — o padrão sugere ser a senha real de produção.

    Se ADMIN_PASSWORD faltar em qualquer ambiente, o painel inteiro fica acessível com essa senha pública no código-fonte. Precisa ser trocada imediatamente na Vercel e o fallback eliminado do código. Como o repositório está no GitHub, essa senha deve ser considerada comprometida.

  2. Rota de upload sem autenticação no POST. src/app/api/admin/upload/route.js — o handler POST (linhas 6-89) não verifica senha alguma. Qualquer visitante anônimo pode subir arquivos para o bucket público imagens do Supabase Storage, sem limite de tamanho ou whitelist de MIME.

  3. Checkout confia cegamente no preço enviado pelo cliente. src/app/api/orders/route.js:69-76 grava price: item.price direto do payload JSON, sem consultar products.preco. Via DevTools, qualquer visitante pode alterar preço/quantidade antes do POST. Mitigado operacionalmente (confirmação humana por WhatsApp), não tecnicamente.

  4. Anon key do Supabase de produção hardcoded no bundle client, contornando toda a autenticação admin. src/app/links/BiolinkView.js:19-20,104-105 tem fallback literal com URL e anon key JWT completas, e faz fetch/PATCH direto em /rest/v1/biolinks..., sem passar pelas rotas /api/admin/biolinks*. A segurança depende inteiramente de RLS — sem evidência disso para essas tabelas.

  5. RLS ausente ou incompleto em tabelas com dado sensível ou gravação pública.

    • app_settings — criada sem RLS habilitado.
    • customer_notes — CRM interno de clientes, sem política em nenhum script.
    • product_reviews — sem política.
    • biolinks/biolink_blocks — sem política (agravado pelo item anterior).
    • supabase_recipes.sql:25-29 cria política USING (true) WITH CHECK (true) que libera INSERT/UPDATE/DELETE para qualquer requisição.

  6. Sincronização com o ERP Solidcon trafega em HTTP puro, sem TLS. api/sync/prices e api/admin/solidcon chamam http://45.239.193.56:5001/... — IP fixo, sem HTTPS. Preço/estoque em texto claro, suscetível a interceptação.

  7. Senha/token passados via query string em vários pontos, expostos em logs de servidor/proxy/CDN e no header Referer: server.js:51, api/admin/solidcon/route.js:9, e o padrão geral do front-end admin (?auth= em ~20 rotas). Migrar para header Authorization.

  8. .git do repositório com 456 MB, sendo 85% (13.104 arquivos) uma aplicação PHP de terceiros (bio-links/) com vendor/ do Composer inteiro commitado — SDKs completos de AWS, Stripe, Laravel, dumps SQL. Mais bancos SQLite binários, XMLs de export e um test-tailwind.css de 475 KB.

  9. npm run lint está quebrado — falta o pacote typescript como dependência transitiva de @typescript-eslint. Nenhum lint roda hoje, localmente ou em CI.

  10. server.js é código morto que descreve uma sincronização automática que não roda (nenhum script npm o chama). Um dev futuro vai assumir que ela funciona em produção — não funciona. A sincronização real já existe via /api/cron/sync.

  11. Filtro de busca de produtos vulnerável a injeção de sintaxe PostgREST. src/app/api/products/route.js:31 interpola search cru dentro de .or(...). Vírgulas e parênteses têm significado especial na sintaxe de filtro — precisa de escaping antes de interpolar.

3 🛠️ Melhorias de Arquitetura & Código (Média Prioridade)

Autenticação/autorização

  • Verificação de senha admin reimplementada quase identicamente em ~17 arquivos de rota diferentes, cada um com pequenas variações não documentadas. Centralizar em src/lib/auth.js.
  • Modelo de senha única compartilhada (sem sessão, sem expiração, sem log de quem fez o quê) é frágil para equipe com múltiplos vendedores/administradores. Migrar para autenticação real com usuários individuais.

Data fetching / cache

  • Zero ISR/SSG em qualquer rota pública — force-dynamic espalhado em todas as API routes e em receitas/*. Catálogo, categorias e receitas mudam pouco: candidatos naturais a revalidate com revalidação sob demanda.
  • BoutiqueClient.js/AdegaClient.js fazem fetch client-side via useEffect quando poderiam ser Server Components — eliminaria o spinner inicial em duas páginas centrais do site.
  • ProductDetails.js busca reviews via waterfall client-side em vez de paralelo no Server Component pai.

Duplicação de lógica (storefront)

  • Formatação de preço reimplementada em pelo menos 6 arquivos — sem formatPrice() compartilhado.
  • Lógica de bandeira de país do vinho copiada literalmente entre AdegaClient.js e ProductDetails.js.
  • Parsing de pontuação de vinho duplicado 3 vezes (2x no mesmo arquivo).
  • Carrinho via localStorage reimplementado em 4 arquivos, sem hook useCart(). Header.js ainda faz polling de 1500ms redundante.
  • "Card de produto" duplicado em 3 lugares em vez de um componente <ProductCard />.

API/backend

  • Sincronização ERP duplicada quase byte-a-byte entre api/sync/prices e api/admin/solidcon.
  • N+1 sequencial em api/sync e api/cron/sync — gargalo conforme o catálogo cresce.
  • Nenhuma paginação em /api/products, /api/admin/orders, /api/admin/customers.
  • Sem índices confirmados em products.sku, products.status, orders.customer_whatsapp no schema Postgres atual.
  • Mensagens de erro vazando error.message do Postgres em ~10 rotas.

Frontend geral

  • HomeClient.js (GSAP + parallax) não é importado por nenhuma rota — código morto carregando GSAP inteiro no dependency tree.
  • Nenhum uso de prefers-reduced-motion em todo o código.
  • InteractiveIngredients.js manipula DOM imperativamente em vez de useState.
  • globals.css com 1.747 linhas e 93 cores hex hardcoded fora do sistema de tokens @theme.
  • Fontes customizadas carregadas do domínio WordPress legado em vez de /public/fonts.

4 🎨 Evolução de UX/UI & Experiência Mobile

Sanitização de HTML ausente end-to-end: título/descrição de produtos e receitas gravados sem sanitização, renderizados via dangerouslySetInnerHTML em pelo menos 8 pontos do storefront. Combinado com a auth fraca do admin, a cadeia senha fraca → gravação sem sanitização → XSS armazenado é real, não teórica.

  • Checklist de ingredientes não é acessível: sem role="checkbox"/aria-checked, não operável por teclado (o <li> não é focável).
  • SEO estruturado incompleto: só a home tem JSON-LD. Produto e receita mereceriam Product/Recipe schema. Sem sitemap.xml nem robots.txt gerados.
  • cardapio/page.js define título via document.title no client em vez de generateMetadata — título real ausente para crawlers.
  • Mobile: styling majoritariamente inline com clamp()/flexWrap em vez de breakpoints Tailwind — funciona, mas difícil de auditar por tela.
  • Imagens de cardápio e logo sem loading="lazy", sem width/height — risco de CLS/LCP numa página provavelmente muito acessada via QR code.
  • Consistência visual: "card de produto" com 3 implementações JSX distintas — o mesmo padrão de inconsistência já corrigido no admin (StatCard, Tag, StatusBadge) ainda não foi aplicado ao storefront.

5 💡 Propostas de Inovação (Liberdade Total)

  • Fechar o loop de confiança do checkout: revalidar preço/status do produto no servidor no POST de /api/orders, sinalizando no admin pedidos com preço divergente do catálogo.
  • Sincronização ERP real via Vercel Cron declarado em vercel.json, substituindo o mecanismo hoje simulado/não declarado.
  • Programa de fidelidade / histórico de compras: área "Minha Conta" simples (login por WhatsApp + código) mostrando pedidos anteriores e permitindo "repetir pedido".
  • Zoom de produto com galeria multi-imagem — corte, selo de certificação, rótulo — via tabela product_images.
  • Notificação push real (Web Push ou WhatsApp Business API oficial) substituindo o polling + som do navegador.
  • Rich snippets de Receita + Produto no Google — potencial de tráfego orgânico gratuito para marca regional.
  • Consolidar a "reforma de consistência" do admin para o storefront: <ProductCard />, useCart() e formatPrice() compartilhados.
  • Separar bio-links/ em repositório próprio (ou remover) — reduziria o .git de 456 MB para uma fração disso.

✓ Recomendação de Priorização

Dado o volume de achados, a sugestão prática é começar pelos itens 1, 2, 3 e 4 da seção "Ajustes Imediatos" (as duas senhas hardcoded, a rota de upload aberta e a anon key exposta) — os únicos com exposição de segurança real e ativa — antes de qualquer trabalho de arquitetura ou UX.
